Welcome to this first lecture on reinforcement learning. In this lecture, I will introduce fundamental concepts that will be extensively used in subsequent lectures, making them crucial to understand. Initially, these concepts will be introduced using intuitive examples. Following this, they will be presented more formally within the framework of Markov Decision Processes.

A recurring example throughout this course is the grid-world. A grid-world consists of a grid where a robot navigates. Grid cells can be of different types: accessible (white), forbidden (yellow), or target areas that the robot aims to reach. The grid-world also has boundaries. The robot can move horizontally (left, right) and vertically (up, down) between cells, but not diagonally. This example is chosen for its ease of understanding and illustrative utility.

The primary task is to find an optimal path from an initial location to a target location. This task raises questions, such as how to define a "good" or "bad" path. While optimal policies will be discussed later, intuitively, a good path avoids forbidden areas and unnecessary detours. For instance, wandering aimlessly or attempting to cross grid boundaries (e.g., moving beyond the top or left edge) should be avoided.

Using this example, we introduce the first core concept: **State**. A state describes the agent's status within its environment. In the grid-world, a state refers to the robot's location, labeled as s1, s2, s3, up to s9. For example, s1 corresponds to a specific (x, y) coordinate on the two-dimensional grid. In more complex problems, a state might encompass additional information beyond location, such as speed, acceleration (for a robot), or other relevant environmental parameters.

The collection of all possible states forms the **State Space**. Despite the term "space" potentially sounding complex, it is simply a set. We use the symbol $S$ to represent the state space, which is a collection of all possible states, denoted by curly brackets. The range of indices (e.g., $i$ in $s_i$) can be specified or inferred from context.

The next fundamental concept is **Action**. In each state, an agent can take a series of actions. In our grid-world example, there are five possible actions, denoted a1 to a5: a1 (move up), a2 (move right), a3 (move down), a4 (move left), and a5 (stay still, represented by a circle).

The collection of all possible actions forms the **Action Space**, represented by the set $A$. It is crucial to understand that the action space can be state-dependent, meaning the set of available actions may vary from one state to another. Therefore, we denote it as $A(s_i)$, indicating that the action space is a function of the current state $s_i$.

The next concept is **State Transition**. This refers to the process where an agent moves from one state to another after taking an action. For instance, if the agent is in state s1 and takes action a2 (move right), the subsequent state would be s2. This transition can be represented as: from s1, taking action a2, transitions to s2.

Consider another example: if the agent is in s1 and takes action a1 (move up), the next state is s1 again. This occurs because attempting to move up from s1 encounters a boundary, causing the agent to bounce back to its original position. This further illustrates state transition.

State transition defines the interaction between the agent and its environment. In a simulation or game, these interactions can be defined arbitrarily. For example, hitting a wall when moving upwards could result in bouncing back to s1, bouncing sideways to s4, or even further to s7. The specific outcome is a design choice within the simulation. However, in real-world scenarios, interactions are not arbitrary; a robot in a specific location cannot move to any arbitrary location, thus limiting how interactions can be defined.

Let's examine the forbidden area. Suppose an agent is in s5 and takes action a2 (move right). What would be the next state? Two scenarios are possible:
1.  **Accessible but Punished:** The forbidden area is accessible, but entering it incurs a punishment. In this case, from s5, taking action a2 leads to s6, meaning the agent has entered the forbidden area.
2.  **Physically Inaccessible:** The forbidden area is physically blocked, like a wall, making entry impossible. In this scenario, from s5, taking action a2 would cause the agent to bounce back to s5.
In this course, we adopt the first case, where forbidden areas are accessible but incur a punishment. This approach is more generalized and challenging than the second, as excluding states would simplify policy search. Allowing access to forbidden areas can lead to interesting behaviors; for example, an agent might risk entering a forbidden area to find the shortest path to the target, a behavior we will explore later.

State transitions can be represented in a table, with rows corresponding to states and columns to actions. For example, from state s1, taking action a1 (move up) results in bouncing back to s1. While intuitive, this tabular representation has limitations: it can only describe deterministic cases where action outcomes are certain. It cannot express stochastic outcomes, such as taking action a1 from s1 and potentially ending up in s1, s4, or s7 due to environmental factors.

A more general approach involves using **State Transition Probabilities**. This introduces the concept of probability into our course. Intuitively, if an agent is in s1 and takes action a2 (move right), the next state is s2. This intuition is formalized using conditional probability. (For those unfamiliar with conditional probability, an introduction to basic probability concepts is provided in the appendix of my book; the concept itself is quite intuitive.)
For example, if an agent is in s1 and takes action a2, the probability of moving to s2 is 1. Consequently, the probability of moving to any other state $s_i$ (where $i \neq 2$) is 0. This illustrates how conditional probability describes deterministic state transitions.
Conditional probability is also essential for describing stochastic cases. For instance, if wind is present, attempting to move right from s1 to s2 might result in a 50% chance of landing in s2 and a 50% chance of being pushed to s5. These probabilities are expressed as $P(s_2 | s_1, a_2) = 0.5$ and $P(s_5 | s_1, a_2) = 0.5$. Thus, conditional probability offers a more general framework for describing state transitions.

Next, we introduce a crucial concept unique to reinforcement learning: **Policy**. A policy dictates the actions an agent should take when in a given state. Intuitively, a policy can be represented by arrows within the grid-world, where each arrow (or a circle for staying still) indicates the prescribed action for that state.
A policy enables the generation of paths or trajectories. For an intuitive demonstration, consider starting at a specific location. If the policy (represented by a green arrow) indicates moving right, and according to the state transition probability, the agent moves to the next cell. Then, if the policy directs the agent downward, it moves to the next state, and so on. Upon reaching a state where the policy indicates staying still, the agent stops. Thus, starting from an initial location, the agent follows a specific path determined by the policy and state transitions. This process is intuitive: starting from any state, the policy guides the agent's movement through the environment.

Beyond visual arrows, policies can be described mathematically using conditional probability. In reinforcement learning, the symbol $\pi$ typically denotes a policy, rather than the mathematical constant pi. A policy $\pi$ is a conditional probability distribution that specifies the likelihood of taking any action in any given state.
For example, for state s1, a policy might dictate that the probability of taking action a1 is 0, while the probability of taking action a2 is 1. The probabilities for all other actions would be 0. The sum of probabilities for all possible actions in a given state must always equal 1. This example describes a deterministic policy for state s1, where action a2 is always chosen.
However, policies can also be stochastic, where outcomes are not certain. For instance, a policy might specify a 0.5 probability of moving right and a 0.5 probability of moving down from a particular state. This means the actual action taken could vary, leading to different possible paths. Using conditional probabilities, this is represented as: for state s1, $P(a_2 | s_1) = 0.5$ and $P(a_3 | s_1) = 0.5$. The sum of these probabilities is 1, and the probabilities for all other actions are 0 (and are often omitted, as every action implicitly has a probability, even if zero).

This policy can also be expressed in a tabular form, where each row corresponds to a state and each column to an action. For example, for state s5, if the probability of taking the action to move down is 1, then the policy for s5 is deterministic. Conversely, for state s1, if there is a 0.5 probability for action a2 and a 0.5 probability for action a3, then the policy for s1 is stochastic. This tabular representation is highly general, capable of describing both deterministic and stochastic policies. In programming, such policies are typically represented using arrays or matrices.

To explain how stochastic policies are executed in practice, consider a policy with a 0.5 probability for action a2 and a 0.5 probability for action a3. This is implemented through random sampling. A random value $x$ is sampled from a uniform distribution between 0 and 1. If $x$ falls within the range $[0, 0.5)$, action a2 is chosen. If $x$ falls within the range $[0.5, 1)$, action a3 is chosen. This is the fundamental principle.
